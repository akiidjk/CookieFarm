# Concepts

This page gives a concise, high-level view of the concepts you'll see throughout the CookieFarm documentation. If you're new to adversarial/dynamic CTFs or exploit orchestration, start here and then follow the hands‑on guides in the Getting Started and Exploits sections.

## What is an A/D (Adversarial / Dynamic) CTF?

An A/D CTF is a competition style where teams run services that other teams can attack and defend simultaneously. Unlike jeopardy-style CTFs (static problems you solve once), A/D CTFs are continuous and stateful:

- Targets run continuously and may change over time.
- Flags appear, expire, and get rotated by services under test.
- Teams must run *exploits* repeatedly to capture fresh flags and submit them to a scoreboard.

CookieFarm is designed to automate and orchestrate these repeated exploit attempts, safely capture flags, and reliably submit them on the team's behalf.

## Flags: types and lifecycle

Flags are the core currency of A/D CTFs. Common types:

- Static flags — fixed values embedded in a service (rare in dynamic CTFs).
- Dynamic flags — generated per instance or per time window; require repeated capture.
- Ephemeral flags — valid only for a short interval, often per-round or per-connection.

Typical flag lifecycle:
1. Flag is generated by a target service.
2. An exploit discovers or extracts the flag.
3. The exploit reports the flag to the orchestrator.
4. The orchestrator validates formatting/rate-limits, then submits the flag to the event/scoreboard.
5. The scoreboard accepts/rejects the flag. Accepted flags may be stored for auditing.

Understanding how flags are generated and validated by the event is crucial for writing robust exploits and avoiding false submissions.

## What is an "exploit farm"?

An exploit farm is a coordinated system that runs many exploit jobs continuously against target services. Key components:

- Server (orchestrator): schedules jobs, stores credentials, coordinates submissions, and exposes APIs for clients and UI.
- Clients (runners): execute exploit code (scripts, binaries, containers) on a schedule or in response to events.
- Dashboard / Monitoring: shows run status, success/failure rates, submission statistics, and logs.

CookieFarm implements this model so you can scale from a single runner to many parallel workers while retaining central visibility, credential management, and submission control.

## Exploit lifecycle (create → run → monitor)

1. Author
   - Create a reproducible exploit that can be executed non-interactively.
   - Prefer clear CLI flags for target, credentials, timeouts, and verbosity.
   - Keep side effects isolated and avoid persistence unless intentional.

2. Test
   - Validate locally against a dev instance or containerized target.
   - Use the `--dry-run`/verbose modes and local mocks when available.

3. Package
   - Containerize or bundle dependencies so the runner environment is deterministic.
   - Ensure secrets (API keys, passwords) are obtained from the orchestrator and never checked into source control.

4. Deploy / Run
   - Register the exploit with the server or upload it via the dashboard.
   - Schedule runs (intervals, concurrency limits, teams to target).
   - Observe resource limits and politeness — excessive parallelism can trigger rate limits or bans.

5. Capture & Submit
   - Sanitise captured flags and follow event submission protocols.
   - Implement backoff and retry policies for transient failures.
   - Avoid duplicate or malformed submissions to reduce noise and potential penalties.

6. Monitor & Iterate
   - Track success rates, runtime errors, and flakiness.
   - Improve reliability (timeouts, error handling, headless runs) based on real run data.

See the detailed authoring and running guides in the Exploits section for templates and examples: /docs/exploits/creating.mdx and /docs/exploits/running.mdx

## Submission pipeline & validation

The submission pipeline generally has these stages:
- Local validation: exploit validates format before sending to orchestrator.
- Orchestrator checks: rate-limiting, deduplication, and basic format rules.
- Event/Scoreboard submission: the final accept/reject decision.

Design exploits to:
- Fail fast on malformed outputs.
- Respect orchestrator rate limits.
- Report clear status and error messages so operators can triage failures quickly.

For more on how CookieFarm interacts with APIs and submission endpoints, see /docs/server/api-reference.mdx and /docs/architecture/flag-pipeline.mdx.

## Monitoring, observability & debugging

Good observability is essential:
- Log every run with a correlation ID, target, timestamp, and result.
- Expose metrics: success rate, median runtime, error types, submission acceptance ratio.
- Dashboard should let you filter runs by exploit, target, or team.

When debugging:
- Reproduce the failure locally with identical inputs.
- Capture verbose logs and request/response traces (without leaking secrets).
- Use stored artifacts (screenshots, HTTP responses) when available.

See /docs/dashboard/monitoring.mdx for recommended dashboards and alerts.

## Security, ethics, and team rules

Running automated exploit code has risks. Best practices:
- Never run exploits against systems without explicit authorization.
- Isolate exploit execution (containers / restricted VMs).
- Protect secrets — retrieve them at runtime from the orchestrator, never bake in keys.
- Respect event and network rules (rate limits, allowed ports/protocols).

Follow your competition's rules and the project's contributing guidelines: /docs/contributing/index.mdx

## Best practices & quick tips

- Keep exploits idempotent where possible: retries should not cause cascading side effects.
- Add configurable timeouts and conservative defaults.
- Use structured logs (JSON) so the dashboard and analytics can ingest them.
- Start with a low concurrency during initial runs; ramp up as reliability improves.
- Automate smoke tests for each exploit after packaging.
- Document assumptions (target versions, auth methods) in the exploit metadata.
