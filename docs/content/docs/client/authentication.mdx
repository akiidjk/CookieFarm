# Authentication (ckc)

This page describes common authentication workflows for the `ckc` command-line client: interactive login, environment variables, and where credentials are stored locally.

Notes
- The exact flags and config file location can vary by `ckc` release. Use `ckc --help` for the version you have installed.
- Where we mention "server" it means your CookieFarm server instance (for example `https://cookiefarm.example`).

---

## Overview

`ckc` supports the following common ways to authenticate to a CookieFarm server:

- Interactive username/password login (recommended for interactive use).

---

## Typical interactive login

Most users will authenticate once per machine using an interactive command which prompts for credentials and stores a session token locally.

Example (interactive flow):

```bash
# Typical interactive login (example)
# The exact command and flags for your version of `ckc` may differ — run `ckc login --help or ckc config login --help`.
ckc login -H https://cookiefarm.example -P password -p 8080
```

After successful login, `ckc` will store a session token (inside ~/.config/cookiefarm/session) locally so subsequent `ckc` commands do not require re-authentication.

---

## Configuration file and locations

`ckc` typically stores configuration (server URLs, profiles, and credential references) in a per-user config file. The exact location depends on the platform and the `ckc` implementation; common locations include:

- `~/.config/cookiefarm/client.yml`
- `~/.config/cookiefarm/shared.yml` (available only after the login)

---

## Troubleshooting

- "Invalid credentials" or "Unauthorized (401)"
  - Verify the server URL is correct and reachable.
  - Ensure the password is correct.
  - Check server logs for more detail on why a request was rejected.

---

## Where to get help

- Check `ckc --help` for client-specific flags and exact config locations.
- Consult the server-side docs (dashboard or API) for token creation and scopes.
- If something is unclear or broken in your installed `ckc`, file an issue or open a support request with your project maintainers including:
  - `ckc --version`
  - The exact command you ran
  - Any error output (redact tokens/credentials)
