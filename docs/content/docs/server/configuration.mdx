This page documents the server-side configuration for CookieFarm. There are two configuration to do:

- Environment variables (Where are stored only server related configuration)
- `config.yml` (Where are stored configuration related to the A/D setup)

Both files are required for an intended working


---

## Example .env

Use environment variables for runtime overrides and sensitive items:

```text title=".env"
PORT=8080
PASSWORD=password
CONFIG_FILE=true
DEBUG=false
```

---

## config.yml reference (sample)

Below is a full example `config.yml` that demonstrates the common options and defaults. Adjust sections to match your environment.

```yaml title="config.yml"
configured: true

# Server
server:
  url_flag_checker: "http://localhost:5001/flags"
  team_token: "c882674c6270b8293fff7f4fb5c4c249"
  submit_flag_checker_time: 120
  max_flag_batch_size: 1000
  protocol: "cc_http"
  tick_time: 120
  flag_ttl: 0 # in ticks
  start_time: "2023-10-01T00:00:00Z"
  end_time: "2023-10-31T23:59:59Z"

# Client
shared:
  services:
    CookieService: 8081
    vulnify: 1337
    app-nc: 1338
  range_ip_teams: 9
  format_ip_teams: "10.10.{}.1"
  my_team_id: 1
  regex_flag: "[A-Z0-9]{31}="
  nop_team: 0
  url_flag_ids: "http://localhost:5001/flagIds"
```

## Field descriptions

Below are explanations for every field shown in the example `config.yml`. Types and units are given where relevant, and notes call out security-sensitive fields.

Top-level
- `configured` (boolean): Indicates whether the config has been initialized/validated. Typically `true` in production or fully-configured environments.

server
- `server.url_flag_checker` (string - URL): The HTTP endpoint where the server sends flag submissions or queries related to flags. Example: `http://flag-checker.local/flags`. Must include scheme (http/https).
- `server.team_token` (string - secret): Authentication token the server uses when contacting external flag-checker services or other integrated components. Treat as a secret: do not commit to public repos and prefer injecting from environment/secret store.
- `server.submit_flag_checker_time` (integer - seconds): How often (in seconds) the server attempts to submit flags to the configured `url_flag_checker` or to poll/check flag status. In the example `120` means every 2 minutes.
- `server.max_flag_batch_size` (integer): Maximum number of flags to include in a single submission/batch to the flag-checker. Helps limit request size and memory usage.
- `server.protocol` (string): Protocol or submission format used for interaction with other services (example `cc_http` refers to a CTF-like HTTP submission protocol). Valid values depend on the implementation (e.g., `cc_http`, `custom`, etc.).
- `server.tick_time` (integer - seconds): Duration of one server "tick" in seconds. The scheduling engine (for flag lifecycle, scoring, or exploitation windows) advances in ticks. In the example, one tick = 120 seconds.
- `server.flag_ttl` (integer - ticks): Flag time-to-live expressed in ticks. A value of `0` typically means flags do not expire; any positive integer means flags expire after that many ticks.
- `server.start_time` (string - ISO 8601 timestamp): The official start time for the event or scheduling window. Used to compute offsets or to enable/disable periodic jobs.
- `server.end_time` (string - ISO 8601 timestamp): The official end time for the event or scheduling window. Jobs and submissions are usually disabled after this time.

shared (client-facing / per-team settings)
- `shared.services` (mapping): A map of internal service names to their port numbers (or addresses in other environments). Example entries:
  - `CookieService: 8081` — the CookieService is available on port 8081
  - `vulnify: 1337` — a vulnerable service (name `vulnify`) on port 1337
  - `app-nc: 1338` — another app on port 1338
- `shared.range_ip_teams` (integer): Number of IP addresses each one corresponds to a team. Used with `format_ip_teams` to compute per-team addresses.
- `shared.format_ip_teams` (string - format template): Template used to generate per-team IP addresses. Must include a placeholder like `{}` (or the formatting style expected by your code). Example: `"10.10.{}.1"` with `my_team_id=2` → `10.10.2.1`.
- `shared.my_team_id` (integer): The numerical ID assigned to this server's team (useful for constructing addresses, team-aware behaviour, or reporting).
- `shared.regex_flag` (string - regex): Regular expression used to validate flag strings. Example `"[A-Z0-9]{31}="` matches flags with 31 uppercase alphanumerics followed by `=`.
- `shared.nop_team` (integer): A special team ID reserved for "no-operation" or placeholder traffic. Many setups use `0` to indicate a non-existent or excluded team (e.g., when testing or deliberately skipping submissions).
- `shared.url_flag_ids` (string - URL): A URL endpoint that returns or exposes flag identifiers that the server should fetch or use when generating/submitting flags. Must include scheme and be reachable from the server runtime.

Notes and security recommendations
- Keep secrets (e.g., `server.team_token`) out of checked-in YAML files. Use environment variables, Docker/K8s secrets, or a secret manager.
- Time values: `submit_flag_checker_time` and `tick_time` in this example are seconds; `flag_ttl` is in ticks (multiples of `tick_time`). Confirm units with your deployed server implementation.
- `format_ip_teams` and `range_ip_teams` must match the network allocation strategy used by the contest infra — mismatches can cause runtime address resolution failures.
- `regex_flag` must match the exact format expected by external checkers; a mismatch will cause valid flags to be rejected or invalid flags to be accepted.

## Secrets & secret management

- Do not commit `config.yml` with secrets to a public repository.
- Do not commit `.env` with password to a public repository.
