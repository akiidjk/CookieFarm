# Server — Overview

The CookieFarm server is the central authoritative component in the system. It is responsible for coordinating exploit runs, accepting results, validating and submitting flags, exposing APIs for clients and dashboards, storing state, and providing observability and management endpoints.

This page gives a high-level description of the server: its role in the architecture, main components, operational concerns (security, scaling, backups), and pointers to the detailed configuration and deployment guides.

---

## Responsibilities

At a glance, the server performs the following duties:

- Expose authenticated REST APIs for clients (`ckc`) and the web dashboard.
- Manage exploit run orchestration and scheduling.
- Receive exploit results and manage the flag submission pipeline.
- Persist configuration, user accounts, exploit metadata, and run history.
- Enforce rate limits, quotas, and security policies.
- Provide metrics, logging, and health endpoints for monitoring and alerting.
- Integrate with persistent stores and external services (e.g., Redis, Postgres, object storage, SMTP).

---

## Where the server fits in the system

CookieFarm's runtime consists of three cooperating roles:

- Server — central coordinator, API provider, and stateful backend (this page).
- Client (`ckc`) — CLI used by exploit authors/operators to upload, run, and manage exploits.
- Dashboard — web UI for real-time monitoring, management, and visualizing results.

The server accepts requests from `ckc` and the dashboard, validates/authenticates them, and coordinates exploit execution either directly (via worker processes) or by enqueueing tasks for distributed workers. The server is also the gatekeeper for flag submission: it receives captured flags from exploit runs and orchestrates the submission/validation lifecycle.

## High-level architecture & components

Below are the server's core components and their responsibilities.

- API gateway
  - Exposes REST endpoints for authentication, exploit management, run orchestration, and administration.
  - Performs request authentication/authorization and input validation.
- Scheduler / Orchestrator
  - Decides when and where exploits should run.
  - Enqueues work to worker pools or external runners.
- Worker pool / Runner integration
  - Either in-process worker threads/processes or integration points for remote runners that execute exploits and report results back.
  - Handles retries, timeouts, and resource isolation.
- Submission pipeline
  - Receives flags from exploit runs, formats them, and forwards them to target submission endpoints.
  - Tracks submission status, responses, and error handling / backoff.
- Persistence layer
  - Primary database (Postgres recommended) for users, exploits metadata, runs, and audit logs.
  - Optional caches (Redis) for ephemeral state, rate limiting, and queues.
  - Optional object storage for large artifacts, logs, or exploit packages.
- Authentication & Authorization
  - API tokens, sessions, and role-based access control for users and services.
- Observability stack
  - Structured logs, metrics (Prometheus), and health endpoints.
- Admin UI / Dashboard integration
  - Endpoints and websocket/streaming channels for the dashboard to receive live updates on runs and submissions.

## Typical data flows

1. Operator uploads an exploit via `ckc` (API call to server).
2. Server stores metadata, validates, and makes the exploit available.
3. Scheduler schedules a run (manual or automatic) and dispatches a job to a worker.
4. Worker performs the exploit, captures artifacts and flags, and reports results back to the server.
5. The server routes captured flags into the submission pipeline, which attempts submission and records outcomes.
6. Dashboard subscribes to realtime events or queries the server for run status and logs.

## Configuration

Server configuration is driven by a combination of environment variables and a `config.yml` file (see the server configuration reference for the full list of settings). Important categories:

- Network
  - Bind address, port, TLS settings (certificates).
- Database
  - DSN for Postgres (or other supported DB), pool sizes, migrations settings.
- Queueing / cache
  - Redis URL, queue prefix, timeouts.
- Authentication
  - JWT signing keys, token expiration, OAuth settings (if enabled).
- Submission
  - Default submission endpoint, credentials, per-target rate limits.
- Security
  - CORS policy, allowed hosts, admin IP whitelists, secret rotation.
- Observability
  - Metrics address, log level, log format, Sentry/remote error collectors.
- Storage
  - Object store endpoint, credentials, retention policies for artifacts and logs.

See /docs/server/configuration.mdx for a full, reference-style breakdown of configuration options and examples.

## Security considerations

The server stores sensitive data (API keys, submission credentials, user credentials) and therefore must be operated with security best practices:

- Secrets management
  - Avoid embedding secrets in the repository. Use environment variables or a secrets manager (Vault, AWS Secrets Manager).
  - Rotate keys periodically and support revocation.
- Network security
  - Use TLS for all external API traffic.
  - Limit administrative endpoints to private networks or authenticated users.
- Authentication & access control
  - Use strong authentication for operators and clients.
  - Apply least-privilege RBAC to administrative actions (exploit upload, server shutdown, secret management).
- Input validation & rate limiting
  - Validate all incoming payloads to avoid injection attacks.
  - Implement per-user and per-target rate limits to prevent abuse.
- Auditing
  - Record important actions (exploit uploads, run cancellations, submission credentials changes) with timestamps and actor identities.

Refer to /docs/server/deployment.mdx for production hardening and recommended platform-specific guidance.

## Persistence, backups & migrations

- Use a reliable SQL database (Postgres recommended) for primary state.
- Run schema migrations as part of your deployment pipeline; ensure migration locks and backups are in place before major upgrades.
- Back up the database regularly and test restores.
- For object storage (artifacts/logs), configure lifecycle and retention policies to avoid uncontrolled growth.
- Keep export/import tooling or scripts available to move data between environments (dev/staging/production).

## Observability & troubleshooting

The server exposes mechanisms that make operations and debugging straightforward:

- Health endpoints
  - `/healthz`, `/ready` for quick liveness and readiness checks.
- Metrics
  - Expose Prometheus-compatible metrics for job counts, queue lengths, submission success/failures, latencies.
- Logging
  - Structured logs (JSON) with correlation IDs for runs and submissions.
- Tracing (optional)
  - Instrument long-running flows (e.g., submission pipeline) with distributed tracing when integrated with a tracing backend.
- Diagnostic endpoints
  - Admin-only endpoints for inspecting queues, job states, and worker heartbeats.

When troubleshooting:
- Use correlation IDs from client requests to trace a run end-to-end.
- Reproduce failing submissions locally against the configured submission endpoint.
- Inspect worker logs and server-side error traces for pattern detection.

## Scaling & high availability

Common strategies for production deployments:

- Horizontal scaling
  - Run multiple server instances behind a load balancer for API scale. Ensure sticky sessions are not required (prefer stateless services).
- Shared backing services
  - Use a single, shared Postgres and Redis cluster that all server instances connect to.
- Worker scaling
  - Separate the worker fleet from API servers. Scale workers based on queue depth and CPU/memory needs.
- Leader election & single-writer concerns
  - If some operations must be single-writer (e.g., scheduled submissions), implement leader election (e.g., via Redis or database advisory locks).
- Stateful components
  - Keep the server stateless as much as possible. For stateful responsibilities, prefer well-known distributed systems (Postgres clusters, managed Redis).

See /docs/server/deployment.mdx for example compose and K8s manifests and recommended HA patterns.

---

## Upgrades & rolling deployments

- Run migrations in a backward-compatible manner.
- For schema changes, prefer a two-step migration (add new columns, deploy code that can use either column, then remove old columns later).
- Use health checks to ensure instances are only removed from LB after readiness is lost and current requests finish.
- Maintain a deployment rollback plan and always test upgrades in a staging environment.

---

## Failure modes & common troubleshooting steps

- Database connection failures
  - Check DSN, network policies, and credentials. Ensure the DB accepts connections from the server's network.
- Worker starvation / long queues
  - Inspect queue lengths, worker heartbeats, and resource utilization. Scale workers or increase concurrency limits.
- Flag submission failures
  - Validate submission credentials, target endpoint reachability, and rate-limit responses (apply exponential backoff).
- High latency / timeouts
  - Monitor latencies, inspect slow queries, and ensure the server has sufficient CPU/memory and DB indices are in place.
