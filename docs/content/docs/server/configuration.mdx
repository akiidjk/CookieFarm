# Server configuration

This page documents the server-side configuration for CookieFarm. There are two common configuration methods:

- Environment variables (recommended for simple deployments and container / Docker Compose setups)
- `config.yml` (recommended for complex deployments and when you want a single versioned config file)

Both methods are supported; environment variables typically override values in `config.yml`. Use whatever fits your deployment workflow.

---

## Overview

- `.env` is useful for secrets and container env injection.
- `config.yml` holds the canonical configuration structure (server binding, DB, caches, storage, exploit runner behavior, submission pipeline, logging, and feature toggles).
- Keep secrets (DB passwords, JWT secrets, S3 keys) out of VCS. Use a secret manager (Vault, AWS Secrets Manager, etc.) or container orchestrator secret features in production.

---

## Example .env

Use environment variables for runtime overrides and sensitive items:

```CookieFarm/docs/content/docs/server/configuration.mdx#L1-20
# .env (example)
CF_SERVER_HOST=0.0.0.0
CF_SERVER_PORT=8080

# Database (Postgres)
CF_DATABASE_URL=postgres://cf_user:password@db:5432/cookiefarm?sslmode=disable

# Redis (caching / queues)
CF_REDIS_ADDR=redis:6379
CF_REDIS_PASSWORD=

# Storage (S3)
CF_S3_ENDPOINT=https://s3.example.com
CF_S3_ACCESS_KEY=AKIA...
CF_S3_SECRET_KEY=...

# Authentication & security
CF_JWT_SECRET=replace-with-a-secure-random-string
CF_ADMIN_TOKEN=replace-with-admin-token

# Misc
CF_LOG_LEVEL=info
```

Notes:
- Use a long, unguessable `CF_JWT_SECRET` for signing tokens.
- `CF_ADMIN_TOKEN` is a convenience admin auth token — prefer role-based auth and short-lived credentials in production.

## config.yml reference (sample)

Below is a full example `config.yml` that demonstrates the common options and defaults. Adjust sections to match your environment.

```CookieFarm/docs/content/docs/server/configuration.mdx#L21-220
# config.yml (example)
server:
  host: "0.0.0.0"
  port: 8080
  base_url: "https://docs.example.com" # used for constructing links in emails/web UI
  read_timeout_seconds: 15
  write_timeout_seconds: 15
  shutdown_grace_period_seconds: 30

database:
  driver: "postgres"
  postgres:
    dsn: "postgres://cf_user:password@localhost:5432/cookiefarm?sslmode=disable"
    max_open_conns: 25
    max_idle_conns: 5
    conn_max_lifetime_seconds: 300

cache:
  redis:
    addr: "localhost:6379"
    password: ""
    db: 0
    pool_size: 10

storage:
  type: "local" # options: local | s3
  local:
    path: "/var/lib/cookiefarm/storage"
  s3:
    endpoint: "https://s3.amazonaws.com"
    bucket: "cookiefarm-storage"
    region: "us-east-1"
    access_key: ""
    secret_key: ""
    use_path_style: false

auth:
  jwt_secret: "REPLACE_ME" # keep secret, override from env
  jwt_expiry_seconds: 3600
  admin_tokens:
    - "one-admin-token"

api:
  enable_openapi: true
  rate_limit_per_minute: 120
  cors:
    allowed_origins:
      - "https://dashboard.example.com"
    allow_credentials: true
    allowed_headers:
      - "Authorization"
      - "Content-Type"

exploits:
  runner:
    concurrency: 10
    default_timeout_seconds: 30
    working_dir: "/var/lib/cookiefarm/exploits"
  uploads:
    max_size_bytes: 10485760 # 10 MB
    allowed_extensions:
      - ".py"
      - ".zip"
      - ".tar.gz"

submission:
  endpoint: "https://submission.example.com/submit"
  timeout_seconds: 10
  retry:
    attempts: 3
    backoff_seconds: 2

monitoring:
  prometheus:
    enabled: true
    listen_addr: "0.0.0.0:9090"

logging:
  level: "info" # debug | info | warn | error
  format: "json" # json | text
  output: "stdout"

security:
  tls:
    enabled: false
    cert_file: ""
    key_file: ""
  allowed_ips: [] # whitelist CIDR blocks (optional)
```

## Field descriptions

- `server.host`, `server.port`: binding host and port for the HTTP server.
- `server.base_url`: used to create absolute links in notifications and the dashboard.
- `database.postgres.dsn`: standard Postgres DSN. Use env var to keep password secret.
- `cache.redis`: Redis used for caching, queues, or rate-limiting.
- `storage.type`: `local` stores files on disk; `s3` stores artifacts in S3-compatible object storage. Provide credentials securely.
- `auth.jwt_secret`: signing secret for JWT tokens. Rotate periodically and store securely.
- `api.rate_limit_per_minute`: global rate limit. You may want per-route overrides in advanced setups.
- `exploits.runner.concurrency`: max number of concurrent exploit runner workers.
- `submission.retry`: controls submission pipeline retry behaviour on transient errors.
- `monitoring.prometheus`: expose metrics endpoint for Prometheus scrapings.
- `logging`: choose `json` for structured logs in production.

## Environment variable overrides

When both `.env` and `config.yml` are present, environment variables override the equivalent `config.yml` values. Use clear env var naming and document any mapping in your deployment repository.

Common env var names (conventions used by CookieFarm deployments):

- `CF_SERVER_HOST`, `CF_SERVER_PORT`
- `CF_DATABASE_URL`
- `CF_REDIS_ADDR`, `CF_REDIS_PASSWORD`
- `CF_S3_ENDPOINT`, `CF_S3_ACCESS_KEY`, `CF_S3_SECRET_KEY`
- `CF_JWT_SECRET`
- `CF_LOG_LEVEL`

## Secrets & secret management

- Do not commit `config.yml` with secrets to a public repository.
- Use your platform's secret-management facilities:
  - Docker Compose: use `secrets:` and an external secret file.
  - Kubernetes: store secrets in `Secret` objects and mount as env vars.
  - Cloud: use managed secret stores and inject at deploy time.
- Prefer role-based credentials with limited scope.

---

## Production recommendations

- Always enable TLS (terminate at a reverse proxy or load balancer).
- Run the server behind a reverse proxy (nginx, Traefik) for TLS, path-based routing, and static asset caching.
- Use a managed Postgres instance with automated backups.
- Run Redis and S3 storage in HA configurations.
- Use structured log aggregation and configure `logging.format: json` for easier ingestion.
- Limit exploit runner concurrency to match CPU/RAM capacity; avoid resource exhaustion.

---

## Validation and schema

- Validate your `config.yml` with a YAML linter (`yamllint`) and, if available, against any provided config schema in the repository (e.g., `config.schema.json` or similar).
- For secret/credential testing, ensure services are reachable from the same network namespace as the server before starting (DB, Redis, S3, submission endpoint).

---

## Troubleshooting

- Server fails to bind: verify `server.port` and that no other process is using the port.
- DB connection errors: check `database.postgres.dsn`, network connectivity, and that the Postgres user has appropriate permissions.
- Exploit runner failures: increase logging to `debug` and inspect `exploits.runner.working_dir` permissions and disk space.
- Missing storage uploads: validate `storage` configuration and S3 permissions (bucket policy / IAM).
