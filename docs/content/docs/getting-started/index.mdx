# Getting Started

Welcome to CookieFarm. This section walks you through what CookieFarm is, the minimum requirements to run it, and a short quick-start to get a local instance up and running so you can log in and run your first exploit.

If you want a short checklist, skip to the Quick start section below. For deeper background read the Concepts page.

---

## What is CookieFarm?

CookieFarm is an exploit orchestration framework for Capture The Flag (CTF) style automation. It provides:

- A server component that stores configuration, schedules exploit runs, and handles flag submission.
- A client CLI (`ckc`) used by exploit runners to register, fetch work, and report results.
- Optional dashboard for visualizing results and managing exploits.
- A lightweight Python helper library and patterns for writing exploit runners.

Read more about the ideas and lifecycle on the Concepts page (link below).

## Prerequisites (high-level)

Before you start, make sure you have the following installed:

- Python 3.8+ (Python 3.10+ recommended)
- Docker & `docker-compose` (or `docker compose`) for local deployments
- Git (to clone the repository)
- A POSIX-like shell (Linux / macOS recommended; WSL on Windows)

For full details and exact versions see the Prerequisites page:
- ./prerequisites.mdx

Quick tip: create a virtual environment for Python work:

```bash
python -m venv .venv
. .venv/bin/activate
pip install -U pip setuptools
```

## Quick start (TL;DR)

This minimal flow gets a server running locally, installs the client CLI, and runs a simple exploit.

1. Clone the repo and change into it:

```bash
git clone https://github.com/your-org/CookieFarm.git
cd CookieFarm
```

2. Start the server (example using Docker Compose):

```bash
cd deployment  # or the repo dir containing docker-compose.yml
docker compose up -d
```

3. Install the client CLI (`ckc`) into your Python environment:

```bash
pip install -e ./client  # or `pip install cookiefarm` if available on PyPI
```

4. Authenticate the CLI against the server:

```bash
ckc login --server http://localhost:8080 --username alice
# follow prompted flow to obtain token / set config
```

5. Run an example exploit (replace with the exploit name you have):

```bash
ckc exploit run example-exploit --targets targets.txt
```

If the server and client are communicating correctly you should see run logs, and flags will be submitted through the server pipeline.

## Concepts (short)

Some core concepts you'll encounter:

- Exploit runner: a program (usually Python) that attempts to capture flags from a target.
- Exploit lifecycle: create → register/upload → schedule/run → capture → submit → validate.
- Flag pipeline: capture by runners → submit via server → validation and scoreboard updates.
- A/D CTF model: Automation / Defense style of running exploits at scale.

## Troubleshooting (common quick checks)

- Can't reach server: check `docker compose ps` and server logs (`docker compose logs server`).
- Authentication failing: ensure the server base URL and API token are correct in `ckc` config (`ckc config view`).
- Exploit not running: confirm the runner has network access to targets and correct Python deps installed.
- Flags not submitted: inspect server submission logs and the architecture `flag-pipeline` page.

If you're stuck, consult the FAQ or open an issue in the repository.
