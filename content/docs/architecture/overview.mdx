# Architecture Overview

This page provides a high-level view of CookieFarm's architecture: the primary components, their responsibilities, how they communicate, and key operational concerns (scaling, security, observability). ## Goals & constraints

CookieFarm is designed for automated exploit orchestration in CTF-style and adversarial testing environments. Key goals that shaped the architecture:

- Reliability: capture, validate and submit flags reliably under varying network and load conditions.
- Scalability: support many concurrent exploit runs and handle spikes in submission throughput during events.
- Extensibility: pluggable exploit runners, easily extendable APIs, and multiple client implementations.
- Observability: strong logging, metrics and alerting to debug and operate during competitions.
- Security: least privilege for runners, secure storage of credentials and safe handling of untrusted exploit code.

---

## High-level components

1. Server
   - Central orchestrator and API provider.
   - Responsibilities:
     - Manage user accounts, authentication and authorization.
     - Store exploit metadata, schedules, and results.
     - Receive and process flag submissions, validate and forward to the submission pipeline.
     - Provide API endpoints consumed by clients and the dashboard.
   - Typical technologies: compiled backend (Go), relational database for persistent state, message queue for background work.

2. Client (ckc)
   - CLI used by exploit authors/operators.
   - Responsibilities:
     - Authenticate to the server and upload/manage exploits.
     - Start, stop or schedule exploit runs.
     - Stream logs/results to the server or local storage.
   - Typically a Python package or single-binary CLI with configuration for server endpoints and credentials.

3. Exploit Runners (Workers)
   - Isolated processes (or containers) that execute exploit code.
   - Responsibilities:
     - Run exploit logic against target services.
     - Capture candidate flags and produce structured results (stdout, errors, timing, network info).
     - Submit candidates to the server for validation and possible submission.
   - Should run in restricted environments (namespaces, resource limits, seccomp) because exploit code may be untrusted.

4. Dashboard (Web UI)
   - Web application for humans to monitor system state.
   - Responsibilities:
     - Display exploit run status, recent captures, submission history and metrics.
     - Allow upload/management of exploits and scheduling.
     - Provide links to logs and debugging tools.

5. Flag Pipeline
   - A composed flow that starts with capture and ends with submission & validation.
   - Components:
     - Capture: exploit runner detects candidate flag(s).
     - Local validation (optional): runner may perform heuristics before sending to server.
     - Server validation: server checks syntactic validity and deduplication rules.
     - Submission: server sends validated flags to official submission endpoints (or scoreboard APIs).
     - Feedback loop: successful/failed submits are recorded and surfaced via the dashboard.

6. Storage & State
   - Persistent store (SQL) for users, exploits, schedules, and results.
   - Append-only logs or object storage for raw run artifacts (logs, captured artifacts).
   - Short-lived caches for rate-limiting and deduplication windows (Redis or in-memory).

7. Messaging & Background Jobs
   - Asynchronous workers/processes for:
     - Running scheduled exploits.
     - Retry logic for submissions.
     - Long-running processing (e.g., archive processing, analytics).
   - Implemented with a job queue (in-memory for small setups; Redis/RabbitMQ/NSQ for scale).

## Data flows

Below are the most important flows in the system.

Capture → Server → Submission
- Runner captures candidate flag(s).
- Runner sends candidate(s) and metadata to Server API (`POST /api/v1/captures`).
- Server validates format, applies dedupe and rate-limit rules.
- Server enqueues submission tasks to background workers.
- Worker submits to external scoreboard API and records outcome.
- Dashboard and client poll or receive notifications of outcomes.

Client → Server (Exploit lifecycle)
- `ckc exploit create` uploads exploit artifact and metadata to Server.
- Server schedules or registers the exploit for immediate or scheduled runs.
- Runner(s) fetch job definitions or receive tasks via the queue.

Dashboard ↔ Server
- Dashboard consumes server REST APIs and optionally subscribes to real-time events (WebSocket/Server-Sent Events) for live updates.

Logging & Observability
- All components emit structured logs and metrics.
- Logs: persisted to centralized log store (ELK/PL or similar) for retention and search.
- Metrics: exported via Prometheus, dashboards in Grafana. Key metrics: run success/failure, submissions/sec, queue depth, flag throughput.

---

## Communication patterns & protocols

- Primary API: REST over TLS for client ↔ server operations (authentication, upload, control).
- Real-time updates: WebSocket or Server-Sent Events for dashboard live data.
- Worker coordination: message queue for tasks and events (push model) or HTTP polling (pull model) depending on deployment constraints.
- Persistence: SQL (Postgres or MySQL) for authoritative data; Redis for transient state and rate-limiting.

Security practices:
- All traffic must be TLS encrypted.
- Short-lived tokens or API keys for worker authentication; rotate regularly.
- Store secrets (API tokens, scoreboard credentials) encrypted at rest; only decrypt in the worker process when needed.
- Apply network segmentation: runners should have tightly scoped network policies and container isolation.

## Deployment model

Small (local / dev)
- Single server process (API + scheduler), local SQLite or Postgres, a few local workers or single-machine containerized runners.
- Useful for development and quick testing.

Medium / Production
- API servers behind a load balancer (replicated stateless processes).
- Postgres as a managed database (with replicas for read scaling if needed).
- Redis for caching and job queue (or managed queue service).
- Horizontal workers (containerized) orchestrated by Kubernetes or Docker Compose with autoscaling.
- Dashboard served by a web frontend static app and a backing API.

High-availability considerations
- Run multiple API replicas and use a managed DB with automated failover.
- Ensure job queue and broker are highly available.
- Use health checks and graceful shutdown for workers to avoid task duplication.

---

## Scalability & performance

- Horizontal scale: API and worker workers are stateless or effectively stateless, enabling straightforward horizontal scaling.
- Bottlenecks to monitor:
  - Database write contention (optimize schema and use batch inserts where possible).
  - Submission throughput to external scoreboard(s) (throttle and backoff strongly).
  - Queue backlog: provision worker autoscaling to drain spikes.
- Caching: use caching for frequent reads (exploit metadata, dashboard lists) to reduce DB load.

---

## Failure modes & mitigation

- Submission failures (network/timeouts/external rate limits)
  - Use exponential backoff and retry policies.
  - Persist failed submissions for manual review and reprocessing.
- Exploit runner crashes or malicious exploits
  - Use container sandboxing, CPU/memory limits and user namespaces.
  - Run untrusted code in disposable environments and collect artifacts for post-mortem.
- Data loss
  - Regular DB backups and tested recovery procedures.
  - Use append-only storage for raw artifacts and logs.
- High CPU/network during events
  - Autoscale workers; use prioritized queues for critical tasks.

---

## Extensibility & integration points

- Pluggable exploit runners: design a runner protocol (job payload and result schema) so new runners (Python, Go, container wrappers) can be added easily.
- Submission adapters: abstract the submission target behind adapters so you can support multiple scoreboard APIs or custom validators.
- Webhooks / Event system: publish significant events (captures, submissions, alerts) so external systems can integrate.

---

## Operational observability

- Logs: all services should emit structured logs (JSON) with contextual fields: trace_id, run_id, exploit_id, user_id.
- Traces: instrument key flows (capture → submission) with distributed tracing (e.g., OpenTelemetry) so you can trace failures end-to-end.
- Metrics: expose Prometheus metrics for queue sizes, submission latencies, run success rates.
- Alerts: alert on high failure rates, queue backlogs and extreme latencies.
