This document is the authoritative reference for the `ckc` CLI shipped with CookieFarm. It lists global options, main subcommands, common flags and examples for typical workflows (configuration, authentication, exploit authoring & execution, and submissions).

---

## Global usage

General form:

`ckc [GLOBAL FLAGS] <command> [command flags] [arguments...]`


Common global flags
- `-c, --config <path>` — path to the client configuration file (default: `~/.ckc/config.yml`).
- `-s, --server <url>` — override server base URL for this invocation.
- `-v, --verbose` — increase logging verbosity (repeat to increase level).
- `--no-color` — disable colored output (helpful for CI).
- `--timeout <seconds>` — per-request timeout.

To list top-level commands:

    ckc help

To get command-specific help:

    `ckc <command> --help`

## Configuration & authentication

ckc stores client-side config (server, tokens, preferences) in the config file. Use the following commands to manage it.

`ckc config` — Manage configuration

- `ckc config show` — Show the current client configuration.
- `ckc config update` — Update client configuration interactively or via flags (`--host`, `--port`, `--username`, `--https`).
- `ckc config reset` — Reset the client configuration to its default state.
- `ckc config login` — Login to the client (prompts for `--password`).
- `ckc config logout` — Remove client session.

## Exploit commands

The `exploit` command group handles creation, running and management of exploit scripts/runners.

`ckc exploit create` — Generate a new exploit template
- Flags:
  - `-n, --name <name>` — Name of the exploit template.

`ckc exploit list` — List all current running exploits

`ckc exploit remove` — Remove an exploit template
- Flags:
  - `-n, --name <name>` — Name of the exploit template.

`ckc exploit test` — Test the exploit on the NOP team
- Flags:
  - `-e, --exploit <path>` — Path to the exploit file to execute.
  - `-n, --name <name>` — Service name to attack.
  - `-d, --detach` — Run the exploit in the background.
  - `-t, --tick <seconds>` — Interval in seconds between exploit executions.
  - `-T, --thread <count>` — Number of concurrent threads to run the exploit with.

`ckc exploit run` — Attack other teams with an exploit
- Flags:
  - `-e, --exploit <path>` — Path to the exploit file to execute.
  - `-n, --name <name>` — Service name to attack.
  - `-d, --detach` — Run the exploit in the background.
  - `-t, --tick <seconds>` — Interval in seconds between exploit executions.
  - `-T, --thread <count>` — Number of concurrent threads to run the exploit with.
  - `-S, --submit` — Direct submit to the game server.

`ckc exploit stop` — Stop the exploit execution
- Flags:
  - `-p, --pid <pid>` — Pid to the exploit file to stop.

`ckc exploit submit` — Send a new flag to the game server
- Flags:
  - `-f, --flag <flag>` — Flag to submit.
  - `-n, --name <name>` — Service name to attack.
  - `-t, --team <id>` — Team ID.
  - `-m, --message <msg>` — Message to include with the submission.

## Examples & workflows

1) Quick login and run an exploit locally

    ckc config login --host cf.example.com --username alice --password secret
    ckc exploit create --name quick-sqli
    ckc exploit run --exploit ./exploits/quick-sqli

2) Upload an exploit and schedule remote execution

    ckc config login
    ckc exploit run --exploit quick-sqli --thread 10 --tick 300

## Troubleshooting

- "Authentication failed": verify server URL, time sync, and token validity using `ckc config show`.
- "Upload rejected": check the server API reference in documentation for schema changes.
- "Submissions rate-limited": consult server admin and verify you're not reusing the same token across many agents.
