# API Reference

This page documents the CookieFarm server HTTP API. The API is RESTful and versioned under `/api/v1`. It is intended for use by the `ckc` client, dashboard, and third-party integrations.

Base URL
- All endpoints are relative to the server base URL, e.g. `https://cookiefarm.example`
- API prefix: `/api/v1`

OpenAPI
- A machine-readable OpenAPI (Swagger) specification is available at `/openapi.json` and `/openapi.yaml` (if enabled).

Authentication & Authorization

- Primary mechanism: bearer tokens (JWT) issued by the server after login.
  - Clients should include `Authorization: Bearer <token>` for protected endpoints.
- Session cookie variant:
  - The dashboard may use an HTTP-only cookie named `cf_session`.
  - When using cookie sessions, CSRF protection is applied to state-changing endpoints; clients should include `X-CSRF-Token` header when required.
- Roles:
  - `user` - regular exploit runner user.
  - `admin` - can manage users, configuration, and view metrics.
  - `exploiter` - permitted to upload and run exploits (depends on deployment policy).
- Token lifecycle:
  - Tokens have expiry (configurable). Use `/auth/refresh` to refresh short-lived tokens if refresh tokens are enabled.

Common request/response conventions

- All request and response bodies are JSON unless noted otherwise.
- Content-Type header: `application/json`.
- Pagination:
  - Query parameters: `page` (1-based), `per_page` (default 20, max 200).
  - Responses with lists include pagination metadata:
    - `meta: { page, per_page, total_pages, total_items }`
- Timestamps are ISO 8601 in UTC (e.g. `2023-10-03T14:23:00Z`).
- Error format:
  - Standard error response contains HTTP status code and JSON body:
    - `{"error": {"code": "ERR_CODE", "message": "Human readable message", "details": {...}}}`
  - Common HTTP codes:
    - `200 OK`, `201 Created`, `204 No Content`
    - `400 Bad Request` – validation failed
    - `401 Unauthorized` – missing/invalid auth
    - `403 Forbidden` – insufficient privileges
    - `404 Not Found` – resource missing
    - `409 Conflict` – duplicate or conflicting state
    - `422 Unprocessable Entity` – semantic validation errors
    - `429 Too Many Requests` – rate limiting
    - `500 Internal Server Error` – unexpected server error

Rate limiting & throttling
- Per-user and per-IP rate limits apply to API requests (configurable).
- For endpoints that submit flags or trigger mass requests (e.g., exploit run endpoints), stricter rate limits may be enforced.

API Endpoints

Auth

- `POST /api/v1/auth/login`
  - Description: Authenticate user and get a bearer token (and optionally refresh token).
  - Body:
    - `username` (string)
    - `password` (string)
  - Response `201 Created`:
    - `{"token":"<jwt>", "expires_at":"<iso>", "refresh_token":"<opt>"}`
- `POST /api/v1/auth/refresh`
  - Description: Exchange a refresh token for a new access token (if refresh tokens are enabled).
  - Body: `{ "refresh_token": "<token>" }`
  - Response `200 OK`: same shape as login response.
- `POST /api/v1/auth/logout`
  - Description: Invalidate current token / session.
  - Auth: required
  - Response `204 No Content`

Health & status

- `GET /api/v1/health`
  - Description: Lightweight health check. Public endpoint (no auth required by default).
  - Response `200 OK`:
    - `{"status":"ok","uptime_seconds":12345}`

- `GET /api/v1/ready`
  - Description: Readiness probe. Returns non-200 when dependencies are unhealthy.
  - Response `200 OK` or `503 Service Unavailable`

Users & Accounts (Admin / Self)

- `GET /api/v1/users` (admin)
  - Query: `page`, `per_page`, `q` (search)
  - Response: list of users with `meta`

- `GET /api/v1/users/{user_id}`
  - Response: user object with `id`, `username`, `email`, `roles`, `created_at`, `last_login_at`

- `POST /api/v1/users` (admin)
  - Create a new user. Body: `{username,email,password,roles[]}`

- `PATCH /api/v1/users/{user_id}` (admin)
  - Update user metadata or roles.

- `GET /api/v1/me`
  - Returns authenticated user's profile.

Exploits

- `GET /api/v1/exploits`
  - Description: List exploits (with filters).
  - Query params: `owner`, `enabled` (true|false), `sort`, `page`, `per_page`
  - Response: `{data: [...], meta: {...}}`

- `GET /api/v1/exploits/{exploit_id}`
  - Response: exploit metadata and current version info.

- `POST /api/v1/exploits` (auth required; role-permission checked)
  - Description: Upload a new exploit package (zip or tar). Depending on deployment, multipart upload or base64-encoded payload may be supported.
  - Headers:
    - `Content-Type: multipart/form-data` (for file upload) OR `application/json` with encoded payload.
  - Body fields:
    - `name`, `description`, `visibility` (`private|team|public`), `file` (binary)
  - Response `201 Created` with exploit object.

- `PATCH /api/v1/exploits/{exploit_id}`
  - Update exploit metadata (description, enabled/disabled, schedule).

- `DELETE /api/v1/exploits/{exploit_id}`
  - Soft-delete or disable exploit (depends on server configuration). Response `204 No Content`.

Exploit execution / runs

- `POST /api/v1/exploits/{exploit_id}/runs`
  - Description: Start an exploit run (ad-hoc). Returns run ID and status.
  - Body:
    - `targets` (array or target selector), `concurrency`, `timeout_seconds`, `env` (optional env vars)
  - Response `202 Accepted`:
    - `{"run_id":"...", "status":"queued", "queued_at":"..."}`

- `GET /api/v1/runs/{run_id}`
  - Get run status, logs (paginated), metrics, and captured flags list.

- `GET /api/v1/runs/{run_id}/logs`
  - Query: `offset`, `limit`, `follow` (boolean) for streaming support via WebSockets / SSE.

Submissions & Flag Pipeline

- `POST /api/v1/submissions`
  - Description: Submit a flag to the pipeline for validation/forwarding.
  - Body:
    - `flag` (string), `origin` (run_id or exploit_id), `target` (target identifier)
  - Response:
    - `200 OK` when accepted, with `{submission_id, status, received_at}`
    - The pipeline may enqueue validation; status values include `queued`, `validated`, `submitted`, `rejected`, `error`

- `GET /api/v1/submissions/{submission_id}`
  - Query submission result and history.

Metrics & Monitoring (Admin)

- `GET /api/v1/metrics` (admin)
  - Description: Aggregated metrics about runs, flags, submissions, failures. Typically protected.
  - Response: JSON metrics or Prometheus text format (configurable).

Configuration & Server Control (Admin)

- `GET /api/v1/config`
  - Return non-sensitive runtime config (read-only). Admin-only.
- `PATCH /api/v1/config`
  - Update dynamic settings (rate-limits, submission policies). Admin-only and audited.

Webhooks & Callbacks

- `POST /api/v1/hooks`
  - Create webhook subscription (admin or user-scoped).
- `DELETE /api/v1/hooks/{hook_id}`

OpenAPI and Schema Resources

- `/openapi.json` and `/openapi.yaml` — canonical API contract.
- `GET /api/v1/schemas` — returns JSON-Schema fragments for core objects (user, exploit, run, submission) to assist clients.

Notifications & Events

- Server supports event streaming via:
  - Server-Sent Events (SSE) endpoint: `GET /api/v1/events?types=run,submission`
  - WebSocket upgrade at `GET /api/v1/ws` (auth required via token query param or Authorization header)
- Events:
  - `run.started`, `run.completed`, `flag.captured`, `submission.updated`, `exploit.uploaded`, etc.

Examples & Patterns (non-executable examples)

- Authenticate and call protected endpoints:
  - 1) Obtain token from `/api/v1/auth/login`.
  - 2) Include header `Authorization: Bearer <token>` for subsequent requests.
- Pagination example: `GET /api/v1/exploits?page=2&per_page=50`

Security notes

- Always use TLS for API endpoints.
- Rotate tokens and use short TTLs for access tokens; prefer refresh tokens if long-lived sessions are needed.
- Use role-based access control to limit potentially dangerous operations (exploit uploads, runs, configuration changes).
- Audit logs are emitted for admin actions; check `admin.audit` index (where available) for traces of config/user changes.

Versioning & Compatibility

- This document describes the `/api/v1` surface. Additive changes are permitted within a minor/patch release, but breaking changes will be introduced in a major version.
- Clients should use OpenAPI to generate client bindings and detect server capabilities.

Extending & Custom fields

- Many APIs accept an `metadata` map for extensible key/value pairs:
  - Example object snippet: `metadata: { "team": "red", "priority": "high" }`
- Server may include feature toggles indicating extra endpoints or behavior under `GET /api/v1/config` (for client adaptation).

Common error codes and troubleshooting

- ERR_INVALID_CREDENTIALS: invalid username/password.
- ERR_TOKEN_EXPIRED: token expired; refresh or re-login.
- ERR_PERMISSION_DENIED: insufficient role for action.
- ERR_RATE_LIMIT: too many requests (check `Retry-After` header).
- ERR_INVALID_PAYLOAD: request JSON failed validation; `error.details` contains field-level errors.
- When encountering `500 Internal Server Error`, collect request-id header (`X-Request-ID`) and server logs for diagnostics.

- Exploit
  - `id`, `name`, `owner_id`, `version`, `visibility`, `enabled`, `created_at`, `updated_at`, `metadata`
- Run
  - `id`, `exploit_id`, `initiator_id`, `status`, `targets[]`, `started_at`, `finished_at`, `stats` (captures, timeouts, errors)
- Submission
  - `id`, `flag`, `origin`, `status`, `validated_at`, `submitted_at`, `result`, `details`
